    01 / 核心概念

    什么是 Agent Harness？

    Harness 是除了模型之外的所有软件基础设施——负责把模型的智能转化为可靠、可控、可长期运行的实际工作能力。

    Agent = Model (LLM) + Harness (驾驭层)
    模型负责"思考"，Harness 负责一切工程可靠性

    Model

    = CPU

    负责推理与文本生成。Claude、GPT、Gemini 等大语言模型是 Agent 的「大脑」

    Harness

    = 操作系统

    上下文管理、工具调用、状态持久化、权限控制、多Agent协调——模型的「完整运行环境」

    Agent

    = 完整应用

    能在真实世界中稳定运行数天、执行复杂任务、自我验证和恢复的「生产级 AI 助手」

    2026 共识

    模型差距正在缩小，真正拉开差距的是 Harness Engineering（驾驭层工程）。竞争优势已从"谁的模型更强"转向"谁的 Harness 更可靠"。

    02 / 问题背景

    为什么需要 Agent Harness？

    原始 LLM 有四个致命局限，每一个都足以让 Agent 在生产环境中失败。

    🧠

    上下文窗口有限

    长任务（数百步工具调用）容易「失忆」或上下文污染。模型无法自行管理有限的注意力窗口。

    💾

    无持久状态

    一次对话结束就忘记之前所有工作。无法跨会话、跨机器延续任务进度。

    🔧

    无法执行现实操作

    不能直接读写文件、运行代码、调用 API、验证结果。模型只能「说」不能「做」。

    🌀

    容易漂移或提前结束

    长时任务中模型可能半途而废、跳过测试、状态混乱。缺乏外部约束和验证机制。

    Harness 的使命：解决这些模型自身无法解决的问题，让 Agent 在真实世界中稳定工作数天甚至数周，而非只在单次对话里「炫技」。

    03 / 架构总览

    Harness 核心组件全景

    一个完整的 Agent Harness 通常包含六大组件——共同构成 Agent 的「操作系统」。

    🎯

    提示与引导层

    系统提示预设、初始化 Prompt、生命周期钩子（Hooks）。让模型"开机"时就知道规则、目标和行为规范。

    ⚙️

    工具与执行层

    工具调用拦截、执行、结果回填；Bash/Code 执行沙箱。把模型的「想做」变成真实的「做了」。

    📚

    上下文与内存层

    上下文压缩、状态持久化、虚拟文件系统、Git 版本控制、进度日志。跨会话记忆、防止上下文爆炸。

    🗺️

    规划与协调层

    任务分解（Planning）、子 Agent 生成与调度。把大任务拆成可管理的小步，支持多 Agent 协作。

    🛡️

    验证与安全层

    中间件、自动测试、人类审批、Human-in-the-Loop、Guardrails。保证每一步都正确、安全、可审计。

    📁

    状态与持久层

    工作空间、Artifact 存储、会话续接机制。让 Agent「下班后」还能无缝接上工作。

    04 / 运行机制

    Harness 典型工作流程

    长时任务通过五步循环实现可靠执行。Anthropic 正是通过这种设计让 Claude 构建整个生产级 Web 应用。

    1

    初始化（Initializer）

    Harness 用专用 Prompt 启动第一个会话，创建文件系统结构、feature_list.json、Git 仓库、进度日志等。为后续工作建立完整基础设施。

    2

    循环执行（Working Agent）

    每个新会话：读取进度文件 → 挑选一个可验证的小任务 → 执行工具调用 → 运行端到端测试 → Git 提交 → 更新进度 → 保存状态。

    3

    上下文管理

    自动压缩历史、把大输出转存到文件、用 Skills 逐步暴露工具，避免一开始就塞满上下文窗口。

    4

    验证闭环

    每步都强制跑测试。人类可审批、失败自动重试或回滚。生成与评估分离——借鉴 GAN 架构，独立的 Generator 和 Evaluator 确保质量。

    5

    结束 / 交接

    任务完成或需要人工介入时，Harness 负责干净交接。支持 Git 提交、进度摘要、状态序列化。

    05 / 概念辨析

    Harness vs Framework vs SDK

    三个容易混淆的概念，本质区别在于抽象层级和「开箱即用」程度。

    Agent Framework

    = 乐高积木

    如 LangChain、LlamaIndex。提供构建块（工具、记忆、链），需要自己组装成完整系统。灵活但需要大量胶水代码。

    Agent SDK

    = 工具包

    如 Anthropic Agent SDK。更底层的编程接口，提供 API 封装和基础工具。面向开发者构建自定义系统。

    Agent Harness

    = 完整机器人

    如 Claude Code。已经组装好的、有明确观点（opinionated）的运行时系统。直接拿来就能跑长任务实验和生产。关注「工程可靠性」。

    关键区别：Framework 给你零件让你搭，Harness 给你一个已经能跑的、经过生产验证的完整系统。Harness 是 opinionated 的——它对如何正确运行 Agent 有自己的强烈观点。

    07 / 核心引擎

    QueryEngine + ask() 循环

    整个 Harness 的「心脏」。不是一次性对话，而是状态驱动的持续循环。

    QueryEngine.ts — Agent Loop 伪代码

    restored-src/src/QueryEngine.ts
    while (true) { // 多轮 tool-use 循环 normalizeMessagesForAPI(); // 消息标准化 + 上下文组装 assembleToolPool(); // 动态加载工具池 await callClaudeApi(streaming); // 调用 Claude Messages API if (tool_use) { await runPreToolUseHooks(); // 前置钩子 await checkPermissions(); // 权限校验 const result = await tool.call(); // 执行真实操作 appendToolResultToHistory(); // 结果回填上下文 await runPostToolUseHooks(); // 后置钩子 continue; // 继续循环 } break; // 模型不再调工具，结束本轮 }

    Stateful Loop

    不是一问一答，而是持续的状态驱动循环。模型每次调用工具后，结果回填到上下文，驱动下一轮决策。

    Streaming + Tool Delta

    支持流式输出和工具增量调用，实时在 TUI（React Terminal UI）中展示思考过程。

    Multi-Tool 并行

    模型一次可调用多个独立工具，Harness 按依赖关系执行并回填结果，最大化并行效率。

    08 / 内存层

    上下文与内存管理

    解决模型「失忆」和上下文爆炸的核心 Harness 组件。让 Agent 能跨数天、跨机器继续工作。

    Micro-compaction 微压缩

    compact/apiMicrocompact.ts

    • 重复的 ls、cat 等工具输出自动折叠成一行摘要
    • 老消息自动总结，保持在上下文窗口之内
    • 智能识别哪些信息可以安全丢弃

    CLAUDE.md 发现机制

    attachments.ts

    • 启动时自动扫描项目根目录、用户目录、全局目录的 CLAUDE.md
    • 把规则、偏好、领域知识注入上下文
    • 实现「零配置」的项目感知

    Memdir 持久化内存系统

    memdir/memdir.ts

    Memory 目录结构
    ~/.claude/projects/<slug>/memory/ ├── MEMORY.md # 全局索引 (≤200行) ├── user_role.md # 用户角色信息 ├── feedback_*.md # 行为反馈 ├── project_*.md # 项目状态 └── reference_*.md # 外部资源引用

    每个事实单独存成 .md 文件，带 frontmatter 元数据。支持按类型分类（user / feedback / project / reference），实现跨会话的「长期记忆」。

    工具执行层

    把模型的「想做什么」变成「真实做了什么」的桥梁。工具不是裸调用，而是被全生命周期管理。

    统一 Tool 接口

    Tool.ts — buildTool() 每个工具必须实现：

    Tool 接口定义
    buildTool({ name: "FileEditTool", inputSchema: Zod Schema, // 严格类型验证 validateInput: (input) => boolean, // 输入合法性检查 checkPermissions: (input) => Decision, // 权限决策 call: (input) => Result, // 真实执行 renderUI: (result) => JSX, // TUI 渲染 shouldDefer: true, // 延迟加载，节省上下文 })

    关键工具矩阵

    工具功能Harness 价值
    FileEditTool精确字符串替换编辑文件比模型重写整个文件更安全、更精确
    BashTool在沙箱中执行 Shell 命令隔离危险操作，自动分类命令危险等级
    LSPTool语言服务器协议集成提供编译器级别的代码理解
    AgentTool生成子 Agent实现多 Agent Swarm 协作
    Read / Write / Glob / Grep文件系统操作比 Bash 更安全、更可审计的文件访问
    10 / 安全层

    验证与安全层

    Claude Code Harness 最强、最 opinionated 的部分。核心原则：不信任模型，强制在安全沙箱里干活。

    四种权限模式

    ✅ default — 逐项确认 📋 plan — 先只读规划，确认后执行 ⚡ auto — 自动允许安全操作 ⚠️ bypass — 全部放行（仅开发用）

    Permission Decision 流程

    Tool Call
    →
    checkPermissions()
    →
    匹配规则
    →
    Allow
    Ask User
    Deny

    Bash 安全分类器

    bashClassifier 自动判断 Shell 命令危险程度：

    • 安全：ls, cat, git status
    • 需确认：npm install, git push
    • 高危：rm -rf, git push --force

    Plan 模式（HITL）

    经典 Human-in-the-Loop 设计：

    • Agent 先只读探索代码库
    • 生成详细实施计划
    • 用户审批后才执行写操作
    • SandboxManager 隔离高危命令

    生产落地关键：Harness 不是「信任模型」，而是强制「模型只能在安全沙箱里干活」。所有写操作都经过权限检查，高危命令需要人类批准。

    11 / 持久层

    状态持久化与会话管理

    让 Agent 真正「下班后还能继续上班」——跨会话、跨机器的无缝延续。

    💾

    Session 持久化

    每个会话以 JSON 格式存储在 ~/.claude/sessions/<uuid>.json，包含完整对话历史、工具调用记录和状态快照。

    🔄

    Resume / Continue

    支持 --continue（继续上次会话）、--resume（选择历史会话恢复）、/resume 交互式恢复。

    ⏮️

    FileHistorySnapshot

    支持 /rewind 回滚任意历史版本。每次工具执行前自动快照，失败可精确回退。

    🌍

    Teleport 跨机器续接

    OAuth + Git 同步，实现「在笔记本上开始，在服务器上继续」。会话状态通过云端传递。

    Worktree Sessions

    每个任务可以开独立 Git worktree 分支，在隔离环境中工作。实验失败直接丢弃 worktree，不影响主分支。这是 Claude Code 独特的「安全实验」机制。

    12 / 协调层

    多 Agent 协调（Swarm）

    通过文件系统通信实现的轻量级多 Agent 架构。Team Lead 负责任务分解和质量把关。

    核心工具

    工具功能
    TeamCreateTool创建团队，初始化共享任务列表
    TaskCreateTool创建任务，设置依赖关系
    SendMessageToolAgent 间发送消息
    AgentTool生成专用子 Agent（探索/计划/执行）

    通信机制

    • 通过 TeamFile + Mailbox 文件通信
    • 团队配置存储在 ~/.claude/teams/
    • 共享任务列表在 ~/.claude/tasks/
    • Agent 间异步消息传递

    Team Lead 架构

    Swarm 工作流
    Team Lead (主 Agent) ├── TaskCreate: 分解任务 ├── AgentTool: 派遣子 Agent │ ├── Explore Agent # 只读探索 │ ├── Plan Agent # 架构规划 │ ├── Code Agent # 代码实现 │ └── Review Agent # 代码审查 ├── SendMessage: 协调分工 └── TaskUpdate: 跟踪进度

    最佳实践：主 Agent 作为 orchestrator 只负责任务分配和质量把关，所有实施工作推给子 Agent。借鉴 GAN 思路：生成与评估分离。

    13 / 全景对照

    Harness 能力 vs Claude Code 实现

    Claude Code 如何用具体模块填充 Harness 的每一个能力要求。

    Harness 能力Claude Code 实现模块解决的模型原生痛点
    持久状态 Memdir + Session JSON + Teleport 无记忆、会话结束就忘
    上下文管理 Micro-compaction + CLAUDE.md 上下文爆炸
    工具可靠执行 Tool 接口 + Pre/Post Hooks + 沙箱 无法安全执行真实操作
    验证闭环 Permission System + Plan 模式 容易漂移、幻觉
    长时任务协调 askLoop + Multi-Agent Swarm 任务半途而废
    可恢复性 Git Worktree + Rewind + Resume 失败后无法继续
    生成/评估分离 Generator Agent + Evaluator Agent 模型自我评价倾向于肯定自身输出

    核心洞察：所有模型做不到的工程可靠性，都被 Harness 补齐了。这就是为什么 Claude Code 能让模型真正「写完整应用」而非只「生成一段代码」。

    14 / 趋势展望

    2026 行业趋势

    Agent Harness 领域的四个重要方向——从工程实践到学科建设。

    01

    Harness 轻量化

    随着模型能力提升，Harness 逐步「瘦身」——很多曾经靠 Harness 实现的逻辑（规划、简单验证）被模型吸收。Harness 只保留最核心的持久化、安全和协调部分。Anthropic 建议：定期验证每个 Harness 层是否仍然必要。

    02

    Harness Engineering 成为新学科

    像传统软件工程一样，需要设计、测试、迭代 Harness 本身。将出现专门的设计模式、测试方法论和评估框架。工程师角色从「代码实现者」转向「Agent 编排者」。

    03

    轨迹数据竞争

    好的 Harness 能产生高质量的成功/失败轨迹日志，用于继续微调模型，形成正反馈循环。竞争优势从「谁的模型好」转向「谁的轨迹数据好」。

    04

    自然语言 Harness

    用自然语言定义整个驾驭逻辑，降低 Harness 构建门槛。CLAUDE.md、Skills、Hooks 等机制已经在实践「用人话定义 Agent 行为」。

    15 / 实践指南

    Harness 设计最佳实践

    来自 Anthropic 工程博客和生产实践的五条核心原则。

    A

    生成与评估分离

    模型是糟糕的自我 QA——它倾向于赞美自己的输出。借鉴 GAN 架构：用独立的 Generator 和 Evaluator Agent。Generator 构建，Evaluator 按照明确标准打分。

    B

    评估标准塑造输出

    仅仅写下评分标准就能改善 Generator 的初始输出——甚至在评估实际运行之前。清晰的验收标准本身就是最好的 Prompt。

    C

    用结构化 Artifact 维持连续性

    进度日志（claude-progress.txt）、Git 提交、初始化脚本——让 Agent 跨会话无缝衔接。每个会话结束时写下进度摘要，下个会话开始时读取。

    D

    主 Agent 只做协调和审查

    把所有实施工作推给子 Agent，主 Agent（orchestrator）专注任务分配和质量门槛。避免「既当裁判又当运动员」。

    E

    随模型进步移除不必要的层

    每个 Harness 组件都编码了对模型能力不足的假设。定期验证哪些层已不再必要，保持 Harness 精简高效。

    16 / 总结

    核心洞察

    模型决定上限，Harness 决定下限和落地能力。
    没有好的 Harness，再强的模型也只是「聪明但不可靠的实习生」；
    有了优秀的 Harness，即使是中等模型也能变成可靠的生产力工具。

    五个关键要点

    • Agent = Model + Harness，两者缺一不可
    • 2026 竞争焦点从模型转向 Harness 工程
    • 生成与评估分离是质量保证的基石
    • 状态持久化是长时任务的生命线
    • 安全层是生产落地的不可妥协底线
